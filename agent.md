# Agent 工作手册

## 用途

这份文档写给 Codex 和未来的新会话。每次开始做项目决策或修改代码前，都应该先读这里。

这个项目是个人团队的微信小游戏，目标是验证“治愈经营 + 解压收集 + 激励视频广告”的轻量商业闭环。

## 每次新会话先读哪些文件

新会话开始时，按顺序阅读：

1. `product.md`
2. `design.md`
3. `current-progress.md`
4. `agent.md`

如果后续已经有代码，还要先查看和本次任务相关的源文件，再动手修改。

## 当前默认产品假设

除非用户明确改方向，否则默认第一款游戏是：

- 治愈经营 + 解压收集微信小游戏。
- 面向短时间休闲游玩。
- 优先验证留存和激励视频广告行为。
- 适合个人团队开发和维护。
- 当前优先题材是猫狗治愈小院：玩家可偏向猫或狗，通过轻合成用品来喂养、添水、陪玩、整理和装饰。

## 不能破坏的产品原则

- 游戏必须让新玩家很快知道怎么玩。
- 游戏必须 3 秒内进入可操作状态。
- 单次游玩目标必须能在 5 分钟内完成。
- 音效是核心体验，不能当作可有可无的后期补丁。
- 不做 PVP，不做排行榜压力，不做抢资源。
- 每日回访用“每日小惊喜”，不要做压迫式签到。
- 猫狗都要被尊重为可选偏好，不要把玩家强制锁死在单一宠物偏好。
- 用品合成服务于照顾和治愈反馈，不能变成复杂合成大系统。
- 第一版必须保持小范围。
- 激励视频广告必须是可选奖励，不是强迫惩罚。
- 在核心循环验证前，不要加入重系统。
- 第一屏不能变成营销页面。
- 不要在没必要时引入复杂后端。
- 第一版不要依赖大量高成本美术。

## 需要保护的核心功能

后续实现后，重点保护这些功能：

- 主场景可操作逻辑。
- 猫狗偏好选择和照顾对象逻辑。
- 用品合成逻辑。
- 照顾、收集、整理等核心动作逻辑。
- 每日小惊喜逻辑。
- 核心音效触发逻辑。
- 金币奖励逻辑。
- 场景成长或装饰升级进度。
- 存档和读档。
- 激励视频广告奖励流程。
- 非强制提示流程。

修改这些功能时，必须做手动验证；有测试条件时也要加自动测试。

## 修改要谨慎的文件

后续如果这些文件或目录出现，修改要特别谨慎：

- 项目配置文件。
- 微信小游戏配置文件。
- 核心游戏状态文件。
- 存档和读档逻辑。
- 经济数值配置。
- 音效资源和音效触发配置。
- 广告接入代码。
- 数据埋点代码。
- 资源清单和资源加载逻辑。

没有明确理由，不要随意重命名或大规模调整这些文件。

## 工作方式

- 优先小步推进，做出可玩的东西。
- 第一版始终围绕核心循环。
- 有代码后，优先沿用已有项目风格。
- 修改前先读文件内容。
- 避免无关重构。
- 保留用户已有改动。
- 如果工作区已有未提交改动，不要擅自回退，除非用户明确要求。

## 测试要求

涉及代码修改时，至少确认：

- 项目能启动或构建。
- 主游戏界面能显示。
- 3 秒内可以进入可操作状态。
- 玩家可以自然选择照顾猫或狗。
- 核心照顾、收集或整理动作能跑通。
- 用品合成能产出可使用的粮食、餐具、饮水工具或玩具。
- 每局目标能在 5 分钟内完成。
- 基础资源会正确变化。
- 音效会在关键动作触发。
- 没有 PVP、排行榜或竞争入口被误加回来。
- 存档不会丢失进度。
- 广告奖励不会重复发放。

涉及界面修改时，至少检查：

- 小屏手机视口。
- 常规手机视口。
- 文字不溢出。
- 按钮可以正常点击。
- 弹窗不会无意遮住关键游戏信息。

## 变现注意事项

广告和支付相关逻辑必须谨慎：

- 激励视频奖励只能在成功回调后发放。
- 广告失败或用户跳过时不能发奖励。
- 前几分钟不要强制广告。
- 如果后续加入内购，必须先确认微信平台和监管要求。

## 合规注意事项

发布或变现前，需要确认当时最新要求：

- 微信小游戏平台规则。
- 游戏出版和审批要求。
- 隐私政策。
- 用户数据处理。
- 实名认证和防沉迷要求，如适用。
- 广告和支付规则。

这些规则可能变化，正式上线前必须查官方来源。
