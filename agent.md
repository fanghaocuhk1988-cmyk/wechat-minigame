# Agent 工作手册

## 用途

这份文档写给 Codex 和未来的新会话。每次开始做项目决策或修改代码前，都应该先读这里。

这个项目是个人团队的微信小游戏，目标是验证“合成 + 小店经营 + 激励视频广告”的轻量商业闭环。

## 每次新会话先读哪些文件

新会话开始时，按顺序阅读：

1. `product.md`
2. `design.md`
3. `current-progress.md`
4. `agent.md`

如果后续已经有代码，还要先查看和本次任务相关的源文件，再动手修改。

## 当前默认产品假设

除非用户明确改方向，否则默认第一款游戏是：

- 轻量合成 + 小店经营微信小游戏。
- 面向短时间休闲游玩。
- 优先验证留存和激励视频广告行为。
- 适合个人团队开发和维护。

## 不能破坏的产品原则

- 游戏必须让新玩家很快知道怎么玩。
- 第一版必须保持小范围。
- 激励视频广告必须是可选奖励，不是强迫惩罚。
- 在核心循环验证前，不要加入重系统。
- 第一屏不能变成营销页面。
- 不要在没必要时引入复杂后端。
- 第一版不要依赖大量高成本美术。

## 需要保护的核心功能

后续实现后，重点保护这些功能：

- 合成棋盘逻辑。
- 物品生成逻辑。
- 订单完成逻辑。
- 金币奖励逻辑。
- 店铺升级进度。
- 存档和读档。
- 激励视频广告奖励流程。
- 新手引导流程。

修改这些功能时，必须做手动验证；有测试条件时也要加自动测试。

## 修改要谨慎的文件

后续如果这些文件或目录出现，修改要特别谨慎：

- 项目配置文件。
- 微信小游戏配置文件。
- 核心游戏状态文件。
- 存档和读档逻辑。
- 经济数值配置。
- 广告接入代码。
- 数据埋点代码。
- 资源清单和资源加载逻辑。

没有明确理由，不要随意重命名或大规模调整这些文件。

## 工作方式

- 优先小步推进，做出可玩的东西。
- 第一版始终围绕核心循环。
- 有代码后，优先沿用已有项目风格。
- 修改前先读文件内容。
- 避免无关重构。
- 保留用户已有改动。
- 如果工作区已有未提交改动，不要擅自回退，除非用户明确要求。

## 测试要求

涉及代码修改时，至少确认：

- 项目能启动或构建。
- 主游戏界面能显示。
- 合成循环能跑通。
- 订单可以完成。
- 金币会正确变化。
- 存档不会丢失进度。
- 广告奖励不会重复发放。

涉及界面修改时，至少检查：

- 小屏手机视口。
- 常规手机视口。
- 文字不溢出。
- 按钮可以正常点击。
- 弹窗不会无意遮住关键游戏信息。

## 变现注意事项

广告和支付相关逻辑必须谨慎：

- 激励视频奖励只能在成功回调后发放。
- 广告失败或用户跳过时不能发奖励。
- 前几分钟不要强制广告。
- 如果后续加入内购，必须先确认微信平台和监管要求。

## 合规注意事项

发布或变现前，需要确认当时最新要求：

- 微信小游戏平台规则。
- 游戏出版和审批要求。
- 隐私政策。
- 用户数据处理。
- 实名认证和防沉迷要求，如适用。
- 广告和支付规则。

这些规则可能变化，正式上线前必须查官方来源。
